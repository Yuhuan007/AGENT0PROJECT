## 模块一：Context / Performance

### 1.1 首轮长窗口 / 多模态输入，first token 慢，如何从 5-10s 压到 2s？

首 token 慢的本质是 **prefill 阶段**：模型要把整个长 prompt 过一遍 attention 才能吐第一个 token，
长度越长、多模态 patch 越多，prefill 越久。优化思路是「少算、复用、并行、遮掩」：

**A. Prefix Caching（最高性价比）**
- system prompt、工具定义、few-shot 这些每轮不变的前缀，做 KV-Cache 复用。
  vLLM 的 automatic prefix caching、SGLang 的 RadixAttention、各家 API 的 prompt caching 都是干这个的。
- 关键：把稳定内容放前面、易变内容放后面，让前缀命中率最大化。

**B. 多模态：异步预处理 + 缓存 embedding**
- 图片/音频 encoder 前向很贵。用户上传的瞬间就先跑 encoder，把 embedding 缓存好，提问时直接拼。
- 图片分辨率分级：先用低清出首 token，需要细节再补。

**C. 感知层面的「假 2 秒」——体验上最划算**
- 流式 + 占位：先秒回「正在查看你的文档…」的 status token，把「白屏 8 秒」变成「立即有反馈」。
- 推测性预热：用户还在打字时，就用已有上下文预热 KV-Cache。

**D. 检索代替全量塞入**
- 长窗口里大部分内容不相关。用 RAG 只取 top-k，prefill 长度砍一个数量级。代价是可能漏召。

**取舍**：Prefix caching 几乎无损、优先上；RAG 换速度但牺牲上下文完整性；流式占位是零成本体验补丁，应默认做。

### 1.2 单 session 聊了 200 轮，context 快爆了，怎么压缩且保持流畅？

分层记忆 + 滚动摘要：

1. **最近 N 轮原文（working memory）**：最近 5-10 轮逐字保留，绝不压缩（追问几乎总指向最近对话）。
2. **中期摘要（episodic）**：较早对话按话题分段，每段用 LLM 压成结构化摘要，滚动进行。
3. **长期事实（semantic）**：抽取稳定事实（偏好、决定、约束）存 KV 结构，可检索。

**保持流畅的关键**：
- 摘要**异步**做，不放在用户等待的关键路径上。
- 压缩时保留指代锚点（实体名、关键数字、未闭环任务），避免「把刚才那个乘以 2」失去指代。
- 不可逆信息（待办、承诺）单独结构化存储，不被摘要模糊。

**本项目现状**：三级 context 管理（轮次限制 → 单条截断 → 超预算丢最旧 + 占位提示）。
够用但粗糙——是「丢弃」而非「摘要」。升级路径：把第三级的丢弃换成滚动摘要。

## 模块二：Memory

### 2.1 半个月后用户问了以前问过的问题，怎么召回更合理？

这是**长期记忆召回**问题，合理召回不是简单向量搜：

**写入时**（存得好才召得回）
- 对话后异步抽取「值得记的东西」：事实、偏好、结论，而非流水账。每条带元数据：时间、话题、来源、重要度。
- 去重/更新：新旧冲突时更新而非堆叠。

**召回时（多路召回 + 重排）**
1. 语义检索：query embedding 找相似——基础，但会漏。
2. 关键词/实体匹配：BM25 补语义漏召。
3. 时间衰减：近期加权，但用户说「以前」时放宽时间窗。
4. 重排：召回一批后用小模型/规则过滤。

**隐藏点**：用户说「以前问过」，理想反应是体现连续性——「你两周前问过类似的，当时结论是…，有变化吗？」。
所以写入时要把 Q-A 配对存，而非孤立事实。

**取舍**：纯向量召回简单但精度差（尤其中文/实体/时间）；多路 + 重排精度高但成本高。产品上给召回加超时预算，失败则降级为不主动引用。

### 2.2 Agent memory 经典框架、发展趋势、头部玩家

**经典框架谱系**：
- **MemGPT / Letta**：LLM 当 OS，主存（context）+ 外存（DB），模型自主换入换出。
- **Generative Agents（斯坦福小镇）**：记忆流 + 重要度/新近度/相关度三因子召回 + 定期 reflection 聚合洞察。
- **向量库 + RAG 派（LangChain/LlamaIndex）**：最工程化，也最「傻」，纯检索缺结构。
- **Mem0 / Zep**：专门 memory 层，提供抽取、更新、图结构（Zep 用时序知识图谱）、去重。

**发展趋势**：
1. 从「检索」到「结构化 + 反思」：加知识图谱、加 reflection。
2. 读写自主化：让模型自己判断记什么、忘什么。
3. 短期 context 压缩与长期 memory 召回融合调度。
4. 多智能体共享/隔离记忆。

**头部玩家**：ChatGPT Memory（跨会话记事实、用户可编辑）、Claude 的 project/context 管理等。
共同特点：记忆对用户透明可控、保守召回（记错比忘记更伤信任）。
（内部实现多不公开，以上为公开信息推断。）

## 模块三：Task

### 3.1 长程任务中模型会忘掉目标，有哪些方案，优缺点？

目标漂移（goal drift）：随着 context 被工具输出、中间步骤填满，原始目标被稀释。

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| 目标常驻（goal pinning） | 原始目标固定在每轮 prompt 顶部/system，永不压缩 | 简单有效、成本低 | 复杂目标占 context；不解决子任务跑偏 |
| 计划分解 + 清单（plan & todo） | 开局生成 todo，每步勾选对照 | 结构化、可追踪、抗漂移强 | 计划本身可能错；动态环境要重规划 |
| 周期性反思（reflection） | 每 N 步问「我还朝目标走吗？」 | 能自我纠偏 | 增加成本和延迟 |
| 分层 Agent（orchestrator + worker） | 上层守目标做调度，下层执行子任务 | 目标与执行解耦、最抗漂移 | 架构复杂、层间通信开销 |
| 外部状态机/scratchpad | 目标/进度/结论写到 context 外结构化存储 | 不受 context 长度影响 | 需额外存储读写逻辑 |

**实践通常组合**：goal pinning（便宜）+ todo list（结构化）+ 关键节点 reflection（纠偏）。
本项目的 `todo_tool` + trace 已有雏形，但还没做「自动对照目标」的 reflection。

### 3.2 「每天早上 9 点根据昨天聊天做复盘总结」怎么设计？

定时触发的长程任务，拆成几层：

- **触发层**：Cron 定时（9:00）投递复盘任务到队列。处理时区、错过补偿、幂等（别重复生成）。
- **数据层**：对话持久化并带时间索引，按范围检索「昨天该用户所有 session」。
- **处理层**：昨天对话可能超 context，走 map-reduce：每个 session 先摘要，再合成日报。产出结构化（话题/决定/待办/异常/跟进点）。
- **交付闭环**：生成后主动推送（消息/邮件），这是主动式 Agent。结论写回长期 memory 以积累。
- **可靠性**：定时任务要有重试、失败告警、执行日志（本项目 trace 机制可用）；任务状态持久化，进程重启不丢。

## 模块四：Tool / Session Runtime

### 4.1 异步工具：不让用户干等但结果重要，怎么设计执行和完成通知？

核心是把长耗时工具从请求-响应的同步路径里剥离：

**执行模型**
- 工具注册时声明同步/异步。异步工具调用后**立即返回 task_id 和「已提交」状态**，不阻塞对话。
- 异步任务丢到任务队列（Celery/MQ），worker 执行，状态持久化（pending/running/done/failed）。

**完成通知**
- 任务完成后通过事件回调把结果注入回对应 session：
  - 主动推送：直接发「你的 X 任务完成了，结果是…」。
  - 事件唤醒 Agent：把完成事件作为新输入喂给 Agent，让它决定如何呈现或继续下一步。
- 结果要能关联回原始意图（task_id ↔ session ↔ 用户请求）。

**难点**：结果回注时机（用户可能正聊别的）、超时与失败兜底、多任务乱序完成的呈现。

### 4.2 session busy 时用户又发消息 / 异步完成事件也到达，runtime 怎么处理？

核心：**每 session 一个事件队列 + 状态机 + 冲突策略**，单 worker 串行消费，保证同一 session 内不并发（避免 context 竞态）。

**用户又发新消息**（看产品定位）：
- 排队：处理完当前再处理新的。简单但「没反应」。
- 打断（barge-in）：新消息优先，停掉当前生成。体验最好但要能干净中止。
- 合并：新消息是补充则并入当前上下文。
- 多数聊天 Agent 选「可打断」，因为连发消息通常意味修正。

**异步工具完成事件到达 busy session**：
- 不能丢也不宜粗暴打断。入队，空闲了再消费；若正是当前对话在等的结果，则唤醒推进。

**状态机**：`idle → busy → idle`，busy 期间事件进队列带时间戳/优先级，加 `interrupted` 态处理打断清理。
**状态转换要原子**（锁或单线程事件循环），避免竞态。

**本项目现状**：同步串行 runtime，`run()` 一次跑完，无真正 busy 态并发处理。
升级需引入 per-session 事件队列和异步执行循环——这是从「脚本式 Agent」到「服务式 Agent」的关键升级。

## 模块五：Agent Runtime 架构对比

### 5.1 Claude Code 工具输出方式 vs 国内 GLM/豆包 OpenAI-compatible function calling

**OpenAI-compatible function calling（GLM/豆包/通义）**
- 模型输出结构化 `tool_calls`（JSON），API 层解析；工具定义以 JSON Schema 传入。
- 优点：标准化、生态统一、换模型不改解析；JSON 强结构解析可靠；原生并行调用。
- 缺点：被 JSON 格式约束，复杂/嵌套/流式不灵活；schema 要预定义死；模型被训成「填表」，开放式操作别扭。

**Claude Code 的方式**
- 倾向用 XML-like 标签包裹工具调用（如 `<function_calls>` 块），而非纯 JSON 字段。
- 优点：标签对模型更「自然」、容错性好（少了引号/逗号崩溃）；流式解析友好；长文本/代码参数无转义地狱。
- 缺点：非标准、与 OpenAI 生态不通用、需自写解析器、跨模型迁移成本高。

**设计哲学**：OpenAI 路线赌标准化与互操作、牺牲灵活；Claude 路线赌贴合模型生成习惯与鲁棒、牺牲通用。
**本项目印证**：用 `[TOOL_CALL]` 标签 + 兜底解析而非纯 JSON，正是因为小模型（glm-4-flash）输出严格 JSON 常崩，
标签更鲁棒——自发走向了 Claude 那套思路。

### 5.2 Openhands 状态机设计的优点？更优雅的实现？

> Openhands（前 OpenDevin）内部细节我不能完全保证准确，以下基于其公开设计理念。

**优点**（核心是 event stream + agent controller）：
- 事件驱动：所有 action 和 observation 都是事件，流经统一 event stream，状态由事件序列驱动。
- 可回放/可审计：事件流即完整历史，调试复现容易。
- Action/Observation 解耦：agent 只产出 action，runtime 管执行并回灌 observation。
- 状态显式：running/paused/awaiting-user/error/finished 明确，便于暂停恢复与人工介入。

**局限与更优雅的方向**：
- 局限：状态多、转换硬编码时会状态爆炸、难维护。
- 更优雅：
  1. **分层状态机（HSM）**：状态分层，避免平铺爆炸。
  2. **声明式转换表**：转换用配置/DSL 描述而非散落 if-else，可读可测。
  3. **事件溯源 + 纯函数 reducer**：状态 = reduce(事件流)，转换是纯函数，天然可测/可回放/可时间旅行调试（借鉴 Redux/Elm）。
  4. **持久化状态机**：状态与事件流落盘，支持重启恢复（长程任务必需）。
  5. **中断/恢复一等公民**：human-in-the-loop、打断、异步事件作为原生能力而非补丁。

**与模块四呼应**：好的 runtime 状态机本质要优雅处理的就是 4.2 的并发事件——
事件溯源 + 分层状态机 + 单 session 串行消费，能把「busy 时来消息/来异步结果」统一在一个模型里解决。
